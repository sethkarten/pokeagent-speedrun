# Data Persistence Architecture

This document details the data persistence strategies used in the Pokemon Emerald Speedrun codebase, covering runtime caching, backup systems, and long-term analysis data storage.

## Overview

The system employs a **three-tier persistence model** to manage data across the lifecycle of a run:
1. **Runtime Cache**: High-speed, transient storage for active run state.
2. **Backups**: Snapshot-based recovery points for resilience.
3. **Analysis Data**: Structured, long-term storage for post-run analysis and debugging.

## 1. Runtime Cache (`.pokeagent_cache/`)

### Purpose
Acts as the working directory for the active agent run. It stores the immediate state required to resume or continue execution.

### Structure
- **Location**: `.pokeagent_cache/{run_id}/` (Run-specific isolation).
- **Key Files**:
  - `checkpoint.state`: Binary emulator state (mGBA save state).
  - `checkpoint_llm.txt`: LLM conversation history and step count (`log_entries`, `agent_step_count`). **NO metrics** are stored here.
  - `cumulative_metrics.json`: Complete metrics history (tokens, cost, actions, all steps). **Single source of truth** for metrics; loaded separately from checkpoint.
  - `checkpoint_milestones.json`: Tracker for game milestones.
  - `checkpoint_maps.json`: Discovered map data.
  - `knowledge_base.json`: Agent's accumulated knowledge (not used by CLI agents).
  - `frame_cache.json`: Recent frame data.
  - **CLI containerized runs only**: `claude_memory/` (agent memory), `workspace/` (agent workspace; orchestrator writes `.agent_directive.txt` and `.mcp_config.json` here on every launch; `.mcp_config.json` is overwritten and set read-only).
  - **CLI session resume**: `last_cli_session_id` — persisted session ID for `--resume` across runs/restores. Written after each session; included in backups.

### Management
- **Access**: Managed via `utils/run_data_manager.py` (e.g. `get_cache_directory()`, `get_checkpoint_llm_path()`).
- **Isolation**: Each run gets a unique subdirectory based on its ID, preventing collisions between parallel or sequential runs.

## 2. Backups (`backups/`)

### Purpose
Provides recovery points to restore the agent's state in case of crashes, errors, or strategic backtracking.

### Structure
- **Location**: `backups/{run_id}/{timestamp}_{objective_id}_{description}.zip`.
- **Content**: A zipped archive of the entire `.pokeagent_cache/{run_id}/` directory at a specific point in time (usually after completing an objective).

### Management (`utils/backup_manager.py`)
- **Creation**: `create_cache_backup()` is called automatically upon objective completion or termination.
  - **Permission Handling**: Uses a custom zipfile walker that gracefully skips files the host user cannot read (e.g., if a containerized agent created root-owned files by mistake, though UID matching should prevent this). Skipped files are logged as warnings.
- **Restoration**: `restore_cache_from_backup()` rolls back state. Restores metrics from `cumulative_metrics.json` if present; otherwise metrics reset.
- **Credential overwrite on restore**: After restore, `seed_agent_auth()` **always overwrites** credential files in agent memory (not just when missing). This ensures the host's current credentials are used instead of stale backup credentials.
- **Cleanup**: `_cleanup_old_backups()` automatically maintains a window of recent backups (keeping the last 50 runs) to manage disk space.

## 3. Analysis Data (`run_data/`)

### Purpose
Stores structured data for post-run analysis, debugging, and visualization. This data is populated at the end of a run or session.

### Structure (`run_data/{run_id}/`)
- **`prompt_evolution/`**:
  - `llm_traces/llm_log.jsonl`: Complete, raw LLM interaction logs.
  - `trajectories/trajectories.jsonl`: High-level reasoning-action-outcome traces.
- **`end_state/`**:
  - Final snapshots of `game_state`, `map_data`, `frame_cache`, and `videos` (if recording was enabled).
- **`agent_scratch_space/`**:
  - Files generated by the agent (e.g., plans, notes).
  - `completed_objectives.json` and knowledge base exports.
- **`agent_logs/` (CLI Mode)**:
  - `session_{NNN}.jsonl`: Raw stdout/stderr logs from CLI agent sessions.

### Management (`utils/run_data_manager.py`)
- **RunDataManager**: A singleton class that orchestrates data collection.
- **Initialization**: Sets up the directory structure at run start.
- **Finalization**: Copies relevant data from the runtime cache to the `run_data` directory upon shutdown.

## 4. Software Engineering Principles Deviation

**Deprecated Patterns (Technical Debt)**
- **Issue**: The codebase still contains references to the legacy root `.pokeagent_cache/` path as a fallback, alongside the new run-specific subdirectories.
- **Principle**: *Maintainability* / *Clean Code*.
- **Impact**: Confusion about where data is actually stored; potential for bugs if the fallback logic interacts unexpectedly with the new structure.

**File Location Inconsistency**
- **Issue**: LLM logs start in `llm_logs/` during runtime and are later copied to `run_data/`.
- **Principle**: *Single Source of Truth* / *Efficiency*.
- **Impact**: Unnecessary file copying and potential for version skew if the copy fails or is interrupted. Logs should likely be written directly to their final destination or a dedicated log directory.

**Checkpoint Atomicity (Reliability)**
- **Issue**: Checkpoints consist of multiple separate files (`.state`, `.json`, `.txt`). There is no atomic operation to save them all.
- **Principle**: *Atomicity* (ACID).
- **Impact**: A crash during a save operation could leave the checkpoint in an inconsistent state (e.g., emulator state saved, but LLM history not), corrupting the run.

**Persistence Logic Scattering (Cohesion)**
- **Issue**: Logic for saving data is scattered across `RunDataManager`, `LLMLogger`, `BackupManager`, and `EmeraldEmulator`.
- **Principle**: *Cohesion* / *Single Responsibility Principle*.
- **Impact**: Changing how data is stored requires modifying multiple files. A unified persistence layer or service would be cleaner.

**Silent Failures (Error Handling)**
- **Issue**: Some copy/backup operations catch exceptions and log warnings but do not propagate errors or retry.
- **Principle**: *Fail Fast* / *Robustness*.
- **Impact**: A user might believe a backup succeeded when it actually failed, leading to data loss during a restore attempt.
